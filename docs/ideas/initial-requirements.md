# Android向け猫ライブ壁紙アプリ - アイデアメモ

## このドキュメントの位置づけ

このドキュメントは、Android向けライブ壁紙アプリのMVPを定義するためのアイデアメモです。まずは最小構成で成立する実装範囲を整理し、後続の設計と実装の判断材料にします。

## プロダクトビジョン

温かい手描き風の部屋を背景に、グレーのロシアンブルーの猫が部屋の奥で気ままに過ごしているように見えるライブ壁紙を提供する。

**一言で言うと**: 手描き風の室内背景と猫の自然なアニメーションで、端末のホーム画面に静かな生活感を生むAndroidライブ壁紙アプリ

---

## 前提条件

- 対象プラットフォームは Android
- 実装言語は Kotlin
- ライブ壁紙の実装は WallpaperService を使う
- 対象デバイスはスマートフォンを優先し、タブレット対応は初期スコープ外とする
- 画面向きは縦向き・横向きの両方に対応する
- ホーム画面の左右スクロールに追従する壁紙オフセットに対応する
- 初期フェーズは Compose を使わない
- 設定画面は後回しにし、まずは壁紙表示そのものを成立させる
- 猫は最低限「歩く」「止まる」「遊ぶ」の3状態を持つ
- 猫へのタップ反応をMVPに含める
- 背景は昼のみで固定せず、昼夜差分を早い段階で追加できる前提で設計する
- バッテリー消費への配慮をMVPの必須要件とする

## 使用素材

- 背景素材
  - `background_room_day.png`
  - 想定サイズ: 約 1536 x 2752
- 猫・遊び素材
  - `cat_walk_1.png`
  - `cat_walk_2.png`
  - `cat_sit.png`
  - `cat_play_1.png`
  - `cat_play_2.png`
  - `toy_yarn.png`
  - 想定サイズ: 各素材とも約 2816 x 1536
  - 透過済みPNGを前提にする

---

## 目指す体験

1. ホーム画面を開いたとき、部屋の背景の中で猫がさりげなく動いて見える
2. 動きは派手すぎず、眺めていて落ち着くテンポで切り替わる
3. 長時間有効にしても端末の発熱や電池消費が目立ちにくい

## MVPで解決したいこと

### 今回まず成立させる価値

1. **ライブ壁紙として正しく動くこと**
- Android のホーム画面上で安定して表示される
- 画面の表示状態に応じて描画を開始・停止できる

2. **猫に最低限の生活感があること**
- 猫が歩く、止まる、遊ぶをランダムに切り替える
- 同じ動きだけが続かず、少しだけ自然な間がある

3. **最小構成でも拡張しやすいこと**
- 設定画面や追加演出を後から足しやすい構成にする
- MVP時点では責務を分けすぎず、しかし描画処理は独立させる

## 推奨ディレクトリ構成

まずは Android Studio の標準構成に寄せたうえで、ライブ壁紙に必要な責務だけを分ける。

```text
app/
  src/
    main/
      AndroidManifest.xml
      java/com/example/catlivewallpaper/
        wallpaper/
          CatWallpaperService.kt
          CatWallpaperEngine.kt
        render/
          SceneRenderer.kt
          BitmapRepository.kt
        model/
          CatState.kt
          CatSprite.kt
          SceneTheme.kt
          SceneState.kt
        logic/
          CatBehaviorController.kt
          FrameTicker.kt
          TouchReactionController.kt
      res/
        drawable-nodpi/
          background_room_day.png
          background_room_night.png
          cat_walk_1.png
          cat_walk_2.png
          cat_sit.png
          cat_play_1.png
          cat_play_2.png
          toy_yarn.png
        values/
          strings.xml
          colors.xml
      xml/
        wallpaper.xml
```

### 構成の意図

- `wallpaper/`: WallpaperService と Engine を置く。Androidのライフサイクルと接続する層。
- `render/`: Canvas 描画、Bitmap 読み込み、拡大縮小、配置計算を担当する層。
- `model/`: 猫の状態や表示に必要なデータ構造に加え、昼夜テーマなどの表示前提をまとめる層。
- `logic/`: 状態遷移、フレーム更新間隔、タップ反応など描画以外の振る舞いを管理する層。
- `drawable-nodpi/`: ピクセル単位の見た目を保ちたい素材をそのまま置く前提。素材解像度次第で `drawable` への変更は検討する。
- `background_room_night.png`: まだ未用意でも、昼夜差分を早期に入れられる前提で受け皿だけ確保する。

## 実装方針

### 1. 最小構成の基本方針

- まずは `WallpaperService` + `Engine` + `Canvas` 描画で成立させる
- OpenGL や Compose は使わない
- 1匹の猫、1つの背景、1つの遊び道具に限定する
- ユーザー設定と高度な物理演算は後回しにし、タップ反応はMVPに含める

### 2. 描画方式

- `SurfaceHolder` に対して `Canvas` を取得し、背景と猫を順番に描画する
- 背景は毎回全面描画するが、計算コストを抑えるため拡大率と表示位置は初期化時または画面サイズ変更時に再計算する
- 縦向き・横向きの両方で破綻しないよう、背景は画面比率に応じて切り出し位置を変える
- `Engine.onOffsetsChanged()` を使い、ホーム画面の左右スクロール量に応じて背景の表示範囲を水平移動させる
- 猫のスプライト切り替えは状態に応じて2コマ程度から始める
- 昼夜差分は同じ描画パイプラインで背景だけ差し替えられる構成にする

### 3. 猫の状態設計

最低限の状態を `CatState` として持つ。

```text
WALK
IDLE
PLAY
```

- `WALK`: `cat_walk_1.png` と `cat_walk_2.png` を交互に切り替えながら横移動する
- `IDLE`: `cat_sit.png` を表示して一定時間止まる
- `PLAY`: `cat_play_1.png` と `cat_play_2.png` を交互に切り替え、近くに `toy_yarn.png` を表示する

### 4. タップ反応の最小仕様

- `Engine.onTouchEvent()` を使い、タップ位置を受け取る
- MVPでは新しい猫状態を増やさず、タップ時に毛糸玉を一定時間その位置に表示して `PLAY` を優先的に発火させる
- 猫が遠い場合は、タップ位置の方向へ少し歩く、または向き直る程度の軽い反応に留める

### 5. 状態遷移の考え方

- 完全ランダムではなく、短時間で状態が切り替わり続けないよう滞在時間を持たせる
- 初期案としては `IDLE -> WALK -> IDLE -> PLAY` のように候補を持ちつつ、一定確率で分岐させる
- MVPでは猫は画面後方の限られた範囲だけを移動させ、前景との奥行き感は配置で表現する
- タップ直後は通常遷移よりも `PLAY` を優先し、反応が見えることを優先する

### 6. バッテリー消費への配慮

- 常時60fps描画は避け、必要なときだけ再描画する
- 目安として、歩行・遊び中は低めの固定間隔描画、停止中は描画頻度をさらに落とす
- `Engine.onVisibilityChanged()` でホーム画面が非表示になったら更新を止める
- `Engine.onSurfaceChanged()` で縦横切り替え時のレイアウトだけ再計算し、通常フレームでは再利用する
- `Handler` または同等の軽量なスケジューリングで次フレームを予約し、無限ループで描き続けない
- Bitmap は毎フレーム読み込まず、初期化時に読み込んで再利用する
- 画面サイズ変更時以外はスケーリング結果の再計算を避ける
- タップ反応で追加描画が必要な時間も短く保ち、常時高負荷な追従処理は避ける

### 7. 設定画面を後回しにする理由

- MVPで最重要なのは「ライブ壁紙として自然に見えるか」と「消費電力を抑えて安定動作するか」の確認
- 設定項目を先に増やすと、仕様分岐と検証コストが先に膨らむ
- まずは動作固定の1プリセットで成立させ、後から速度、出現頻度、背景切り替えなどを開放する

## MVP機能要件

### P0 (最初に実装する範囲)

#### 1. ライブ壁紙の基本表示

- `background_room_day.png` を壁紙全体に表示する
- 背景の上に猫を1匹表示する
- Android のライブ壁紙として適用できる
- 縦向き・横向きの両方で表示が破綻しない
- ホーム画面の左右スクロールに合わせて背景オフセットが動く

#### 2. 猫の基本アニメーション

- 猫は「歩く」「止まる」「遊ぶ」を持つ
- 歩きは2枚、遊びは2枚、停止は1枚の素材で構成する
- 状態に応じて表示素材と位置を更新する

#### 3. 行動制御

- 猫は一定時間ごとに状態を切り替える
- 歩行時は左右いずれかへゆっくり移動する
- 遊び状態では毛糸玉を表示する

#### 4. タップ反応

- 画面タップ時に猫が反応する
- MVPではタップ位置に毛糸玉を一時表示し、猫が遊び状態へ入りやすくなる
- 追加素材なしで成立する最小反応を優先する

#### 5. 低負荷な更新制御

- ホーム画面表示中のみ描画更新する
- 停止中の描画更新頻度は歩行時より下げる
- 不要なオブジェクト生成を抑える

### P1 (MVP後に検討)

#### 6. 設定画面

- 猫の速度
- 行動頻度
- 壁紙の表示位置やスケール調整

#### 7. インタラクション追加

- 長押しや連続タップへの反応
- 毛糸玉に近づく演出を増やす
- タップ位置までの自然な経路移動

#### 8. 背景バリエーション

- 夜背景（最優先の初期拡張）
- 天候差分
- 季節差分

---

## 推奨実装ステップ

1. `WallpaperService` と `Engine` を用意し、背景だけ表示する
2. 縦向き・横向きとホーム画面オフセットに対応した背景描画を入れる
3. 猫1枚を固定位置に描画し、スケーリングと表示位置を調整する
4. `WALK` と `IDLE` を追加し、フレーム切り替えと移動を入れる
5. `PLAY` と `toy_yarn.png` を追加し、タップ反応を接続する
6. 昼夜差分を入れやすい背景切り替え構造に整える
7. 描画間隔を調整し、可視状態と非可視状態で更新を制御する

---

## 非機能要件

### パフォーマンス

- ホーム画面表示中に過剰なフレーム更新を行わない
- 画像デコードは初期化時にまとめ、毎フレーム処理を避ける
- 普段使いで発熱やバッテリー消費が目立ちにくいことを優先する
- オフセット計算と縦横再レイアウトは必要時のみ行う

### 保守性

- Android依存のライフサイクル処理と、猫の行動ロジックを分離する
- 将来の設定画面追加に備えて、状態や描画設定をモデルとして切り出せる形にする
- 昼夜差分を追加しても描画ロジックを増やしすぎないよう、背景テーマを差し替え可能にする

### 使いやすさ

- MVPでは設定なしでもすぐ見栄えする初期値に固定する
- 壁紙適用直後に違和感なく表示されることを優先する

## 技術的な検討事項

### 推奨する最初の実装選択

- 描画は `Canvas` ベース
- アニメーションはスプライト差し替えベース
- 更新制御は `Handler` などによる間欠的スケジューリング
- 素材管理は `BitmapFactory` で読み込み、メモリ上に保持
- 横スクロールは `onOffsetsChanged()` を起点に背景の切り出し位置を更新する
- タップ反応は `onTouchEvent()` からロジック層へ渡し、描画層に直接分岐を持たせすぎない

### MVPでは見送るもの

- OpenGL / GLSurfaceView ベースの実装
- Compose UI
- 複雑な設定画面
- 多数の猫や多層パララックス
- ネット接続や外部データ保存

## 確認済みの前提

以下はユーザー確認済みの前提であり、設計時の固定条件として扱う。

1. タブレット対応は不要で、スマートフォンの縦向き・横向きに対応する
2. ホーム画面の左右スクロールに合わせた壁紙オフセット対応が必要
3. 猫へのタップ反応はMVPに含める
4. 猫と遊び素材は約 2816 x 1536、透過済みPNGを前提にする
5. 背景は約 1536 x 2752 で、昼夜差分を早めに見込む

## 成功指標

- Android端末で壁紙として適用し、背景と猫のアニメーションが破綻なく表示される
- 猫の3状態が視覚的に区別できる
- 縦向き・横向きの両方で背景配置と猫の位置が破綻しない
- ホーム画面スクロールとタップ反応がMVPとして成立している
- MVP段階で過度な電池消費を招かない設計方針になっている
